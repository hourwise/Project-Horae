# ADR-XXXX: Mandatory Content Preflight Routing in Horae

- **Status:** Proposed
- **Date:** 2026-07-12
- **Decision owners:** Project Horae maintainers
- **Applies to:** Project Horae
- **Depends on:** a future cross-owner Project Adrasteia content-preflight contract

## Context

Horae routes and coordinates work across the Fates ecosystem. Project Adrasteia Stage-A explicitly excludes content preflight, so this proposal is deferred and makes no current compatibility claim.

A legitimate tool may return hostile instructions, secrets, malformed structures, embedded scripts, or excessive payloads. The routing layer is the correct place to prevent bypass.

## Decision

Horae will introduce a mandatory **Content Surface Preflight** stage for covered content-bearing operations.

```text
Content request
    ↓
Source resolution and trust classification
    ↓
Deterministic preflight observation
    ↓
Ananke policy evaluation
    ↓
Approved exposure level
    ↓
Controlled extraction or projection
    ↓
Agent, user, downstream tool, or Mnemosyne
```

Horae coordinates and enforces the flow. The observer reports facts; Ananke decides.

## Covered Inputs

At minimum:

- local and repository files;
- uploaded attachments;
- downloaded documents;
- archives;
- PDFs and office files;
- images with metadata;
- web snapshots;
- tool results containing document bodies;
- content intended for persistent memory;
- content crossing MCP or runtime boundaries.

## Progressive Disclosure

Recommended stages:

1. `observe_surface`
2. `inspect_structure`
3. `inspect_sanitized_metadata`
4. `request_selected_fields`
5. `request_excerpt`
6. `request_full_content`

Each elevation may trigger a new Ananke decision.

## Routing Rules

1. Covered content cannot reach an agent before preflight.
2. Scan failure, unsupported formats, or resource limits do not fall through to full access.
3. Scanner timeout is a failure state.
4. Delivered content must bind to the approved receipt.
5. Source mutation invalidates the receipt.
6. Full exposure requires an explicit `FULL_CONTENT` decision.
7. Selected or redacted content must be generated by a controlled projection step.
8. Detector configuration must remain inaccessible to ordinary agents.
9. Consumers cannot bypass the preflight route.
10. Every elevation is auditable.

## Trust Classification

Suggested classes:

- `RUNTIME_OWNED_TRUSTED`
- `PROJECT_OWNED`
- `USER_PROVIDED`
- `REMOTE_FETCHED`
- `TOOL_RETURNED`
- `THIRD_PARTY_REPOSITORY`
- `UNKNOWN`

Trust classification influences policy but does not replace inspection where content can change outside the trusted boundary.

## Failure Behaviour

The default posture is fail-closed.

```ts
type HoraeContentRouteResult =
  | { status: "ROUTED"; receiptId: string; exposureLevel: ContentExposureLevel }
  | { status: "WAITING_FOR_APPROVAL"; approvalId: string; receiptId: string }
  | { status: "DENIED"; reasonCodes: string[]; receiptId?: string }
  | { status: "STALE_STATE"; previousReceiptId: string }
  | { status: "FAILED"; reasonCodes: string[] };
```

## Consequences

### Positive

- Prevents direct ingestion of untrusted content.
- Makes progressive disclosure reusable across transports.
- Protects Mnemosyne and reduces token use.
- Gives Ananke reliable evidence before decisions.

### Negative

- Adds latency and routing states.
- Requires scanner lifecycle management.
- Trusted-path configuration must be explicit.

## Acceptance Criteria

- All covered routes pass through preflight.
- Source changes invalidate receipts.
- Full content cannot be returned without a matching Ananke decision.
- Failure and timeout states fail closed.
- Integration tests prove direct routes cannot bypass preflight.
- Audit output links source, observation, decision, projection, and consumer.
